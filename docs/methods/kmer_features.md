# K-mer feature pipeline

> Companion figure: `docs/figures/kmer_method_overview.png` (+ `.pdf`).
> Script to regenerate: `src/analysis/plot_kmer_method.py`.
> This doc is draft methods text for `paper_outline_v2.md` §2.x.

## Overview

For each genomic segment (one DNA contig per influenza segment), we compute
a **bag-of-k-mers** frequency representation. Each segment becomes a fixed
4,096-dimensional vector whose i-th entry is the number of times the i-th
lexicographically-ordered 6-mer occurs in the segment's DNA. Stacked across
all segments in the dataset, this yields a sparse matrix of shape
(`N_segments`, `4^k`) which is the feature cache for the pair classifier.

Pair features for classification are formed by looking up the two segments'
k-mer vectors and applying a fixed interaction (concat, diff, unit_diff, or
element-wise product) before passing to an MLP.

The production configuration is **k = 6**, **no normalization** (raw integer
counts), **all overlapping windows**, **ACGT-only alphabet** (skip any window
containing an ambiguous base).

## GTO → contigs

A Genome Typing Object (GTO) is a JSON document with two relevant arrays:
`contigs[]` (one entry per genomic segment, with keys `id`, `replicon_type`,
`dna`, `length`) and `features[]` (one entry per CDS, with
`protein_translation`). For the k-mer pipeline only `contigs[]` is used.
Stage 1 emits one row per contig to `genome_final.csv`, preserving
`(assembly_id, genbank_ctg_id, canonical_segment, dna_seq)` plus summary
statistics (length, GC content, ambiguous-base counts).

## K-mer counting

For a DNA string `s` and window size `k`, the count vector is built by
sliding a width-`k` window with step 1 across `s`. A window is recorded only
if every character is in `{A, C, G, T}` (uppercase, after `.upper()`).
Windows containing any ambiguous base (N, R, Y, W, S, K, M, B, D, H, V,
gaps) are silently skipped. No reverse-complement canonicalization, no
strand awareness.

```python
def compute_kmer_counts(seq, k):
    counts = Counter()
    for i in range(len(seq) - k + 1):
        kmer = seq[i:i+k].upper()
        if set(kmer) <= {'A','C','G','T'}:
            counts[kmer] += 1
    return counts
```

The output vector has dimension `4^k` (k=6 → 4,096). Columns are ordered
lexicographically: index 0 is `AAAAAA`, index 1 is `AAAAAC`, …, index 4,095
is `TTTTTT`.

Counts are stored **unnormalized** in the production cache. Length
normalization (`l1` → frequencies) and magnitude normalization (`l2` → unit
vectors) are supported config options (`kmer.normalize`) but have not been
used in paper experiments to date; the MLP + `slot_norm` combination
appears to handle raw counts well (see `roadmap_v2.md` Runtime Analysis).

## Sparse storage

The per-segment vectors stack into a single `scipy.sparse.csr_matrix` saved
as `kmer_features_k6.npz`. A sidecar parquet file
(`kmer_features_k6_index.parquet`) maps `(assembly_id, genbank_ctg_id,
canonical_segment)` tuples to row indices. CSR format gives O(1) dense-row
extraction for batch k-mer lookup during training. The total on-disk
footprint on the full dataset is ~6 GB.

## Pair feature construction

For a protein pair (`protein_a`, `protein_b`) generated by Stage 3, the
segment identities are `(assembly_id_a, ctg_a)` and `(assembly_id_b, ctg_b)`
respectively. At train time each pair row is converted to:

1. row_a = index_lookup(assembly_id_a, ctg_a)
2. row_b = index_lookup(assembly_id_b, ctg_b)
3. v_a = kmer_matrix[row_a].todense()  (4,096-dim)
4. v_b = kmer_matrix[row_b].todense()  (4,096-dim)
5. pair_feature = f(v_a, v_b)

Supported interactions `f(·, ·)` (from `get_kmer_pair_features`):

| Interaction | Formula | Output dim |
|---|---|---:|
| `concat` | `[v_a ; v_b]` | 8,192 |
| `diff` | `|v_a − v_b|` | 4,096 |
| `unit_diff` | `(v_a − v_b) / ‖v_a − v_b‖_2` | 4,096 |
| `prod` | `v_a ⊙ v_b` (element-wise) | 4,096 |

The paper's 28-pair sweeps use `concat` + `slot_norm` (LayerNorm per slot
before concatenation); see `paper_outline_v2.md`.

## Anticipated reviewer questions

These are likely questions with short answers. Some are already covered by
roadmap/audit docs; cross-references given.

### Why k-mers at all?

The task is *segment co-occurrence*: given two genomic segments, decide
whether they come from the same isolate. A bag-of-k-mers representation
captures compositional signal (GC content, codon bias, di/trinucleotide
usage) without requiring alignment, a reference, or a learned embedding
model. It is:

- deterministic and cheap to compute (CPU-only);
- interpretable (each feature is a specific 6-mer count);
- length-tolerant (works across 838–2,341 bp segments without padding);
- independent of gene/protein annotation (works on raw DNA).

For influenza A, where each segment codes for a small number of proteins
with characteristic codon-usage signatures, composition alone is a strong
segment-identity cue — which is why the paper's k-mer MLP matches or
exceeds ESM-2 protein-embedding baselines on the full 28-pair sweep
(k-mer median AUC 0.994 vs ESM-2 0.976; see `roadmap_v2.md` §11).

### Why k = 6?

k = 6 is the smallest k at which the 4^k = 4,096 vocabulary is richer than
the longest segment (~2,300 bp → ~2,295 overlapping 6-mers), so an
individual 6-mer count is informative rather than saturated. k = 4 (256
dims) is too coarse (most 4-mers appear in every segment, count variance
dominated by length). k = 8 (65,536 dims) inflates feature dim by 16× with
diminishing biological discriminability per bit. k = 10 (1 M dims) is
tractable in sparse form but would dominate MLP parameter count. We did
not tune k exhaustively; k = 6 is the standard choice in k-mer-based
virology papers and it works out of the box here.

### Why not normalize counts?

Raw counts encode both composition *and* length. Since segment lengths
differ substantially (S1 at 2.3 kb vs S8 at 0.9 kb), raw counts carry
some length information as a confound. Two responses:

1. We apply per-slot LayerNorm (`slot_transform: slot_norm`) in the MLP
   before the interaction feature is computed. Empirically this has been
   the critical ingredient for ESM-2 embeddings and also helps k-mer
   performance on homogeneous subsets.
2. We have not run a head-to-head comparison of `l1`-normalized k-mers
   vs raw + slot_norm on the full 28-pair sweep. Possible future ablation.

For paper text: state that counts are raw, slot_norm is applied downstream;
flag the length-confound as a known property and an ablation candidate.

### What about ambiguous bases (N, R, Y, …)?

All windows containing any non-`ACGT` character are **skipped**, not
imputed. For the full Flu-A July 2025 dataset this removes a negligible
fraction of windows (flu reference genomes are mostly unambiguous; the
`ambig_frac` column in `genome_final.csv` captures the per-segment rate
and is near 0 for well-assembled sequences). A segment with pathological
ambiguity would end up with a partial k-mer profile; the existing Stage 1
QC (drops `genome_missing_seqs.csv`, `genome_poor_quality.csv`) filters
those out before Stage 2b runs.

### What about reverse complement?

No. The k-mer counter operates on the DNA string as stored in the GTO,
which is the sense strand for each contig. We do not sum counts of each
k-mer with its reverse complement. For segment co-occurrence this matches
the input encoding the protein-based ESM-2 pipeline also uses (each
segment has one canonical sequence per GTO). Reverse-complement
canonicalization would halve the effective vocabulary (to ~2,080 unique
canonical 6-mers for odd-k, or similar) and could be tried as an
ablation.

### Are k-mers position-aware?

No — the representation is a multiset. Two sequences with the same 6-mer
composition but different arrangements will produce identical feature
vectors. For this task that's acceptable: flu segments are compact enough
that compositional difference between segments (e.g., PB2 vs HA) is
already strong, and compositional difference between isolates of the
same segment is a direct proxy for evolutionary divergence.

### How does the k-mer vector link to a protein pair?

Stage 3 produces pair rows with `(assembly_id_a, ctg_a)` and
`(assembly_id_b, ctg_b)` where `ctg_x` is `genbank_ctg_id` — the same key
used to index the k-mer matrix. The training dataloader
(`KmerPairDataset`) builds a composite key
`f"{assembly_id}::{genbank_ctg_id}"` and looks up both sides of each pair
via the parquet index.

### Does this introduce leakage?

Two sub-questions:

1. *Can the same k-mer vector end up in train and test?* Only if two rows
   share `(assembly_id, genbank_ctg_id)`, which Stage 3's
   isolate-partitioned split already prevents (all contigs from a given
   isolate go to one split).
2. *Can semantically-identical k-mer vectors end up in different splits
   via different `genbank_ctg_id` values?* Yes, in principle — two
   isolates with identical DNA for a given segment produce identical
   k-mer vectors. Stage 3's `pair_key` dedup catches this at the protein
   level (via `seq_hash` from `md5(prot_seq)`), which is a superset of
   what a DNA-hash dedup would catch in the 28-pair setup (since pair
   bundles avoid proteins that share a contig). See
   `docs/plans/done/subtype_balancing_plan.md` and
   `src/analysis/audit_split_leakage.py` for the explicit audit.

### Why does the matrix have 868K rows for a ~111K-isolate dataset?

~111K isolates × 8 segments ≈ 888K segments; after Stage 1 QC (dropping
contigs with missing DNA, poor quality, or unassigned replicon_type)
868,240 remain. The matrix is keyed by segment, not by protein — multiple
proteins can share one segment (M1/M2 on S7; NS1/NEP on S8; PA/PA-X on
S3). Those proteins will receive **identical** k-mer vectors when looked
up, which is semantically correct (they share DNA) and is why the
28-pair Task 11 bundles select exactly one function per segment.

### How does this compare to an alignment- or reference-based approach?

The dataset spans many subtypes (H1-H18, N1-N11) across 2015-2024,
including reassortants. A single reference would force all sequences to
be aligned against one donor segment per position, which (a) favors
strains close to the reference, (b) doesn't generalize to new subtypes,
and (c) requires a multiple-sequence-alignment pipeline per segment
type. The k-mer pipeline is reference-free and treats every segment
uniformly regardless of type.

### Does it scale?

Yes. Stage 2b is CPU-only, linear in total DNA length, and single-pass.
Full-dataset runtime on Lambda is ~5–10 minutes on one CPU for the
~1.4 Gb of total DNA. Output is ~6 GB uncompressed sparse. k = 10 is the
practical ceiling (1 M-dim, ~4 GB sparse with similar nnz).

### Is this common practice?

K-mer feature matrices + linear or MLP classifiers are standard in
metagenomic classification (Kraken, centrifuge use k-mer indexes for
taxonomic assignment; sourmash uses MinHash over k-mers for containment
estimation). The novelty in this paper is not the featurization itself
but the **task** (pair-wise segment co-occurrence) and the head-to-head
comparison against frozen protein-language-model embeddings on the same
pair task.
