## §1. The task in one slide (BIO + ML)

Binary classification on viral protein **pairs**: given two protein
segments, predict whether they co-occur in the same isolate.

- **Positive pair:** `assembly_id_a == assembly_id_b`.
- **Negative pair:** `assembly_id_a != assembly_id_b`.

Each experiment fixes a **schema_pair**, e.g.
`("HA", "NA")` for HA/NA or
`("PB2", "PB1")` for PB2/PB1. Slot A is always
`schema_pair[0]`, slot B always `schema_pair[1]`. This removes input
direction as a free variable for directed pair features.

Primary virus: **Influenza A**. The 8-segment Flu A genome admits
`C(8,2) = 28` schema pairs.

**Why this question is biologically interesting.** A model that scores
arbitrary segment pairs for "do these belong together?" supports
database remidiation (flag mis-joined records), wastewater surveillance
(assemble fragmented metagenomic data), and reassortment forecasting
(segment pairs the model rates as compatible but never co-observed are
candidate future reassortants worth monitoring).

[!] The model is trained to discriminate at the **isolate** level, not
at the population level. The line between "biology learning" and
"metadata memorization" is the central evaluation question
(see §32).

# Part A — Raw GTO data

## §2. What a GTO is (BIO)

A **Genome Typed Object (GTO)** is a single JSON. One GTO = one assembly = one isolate's
reconstructed genome. For Flu A this means **one isolate with 8
segments — verified 108,530 / 108,530 (100.00%) assemblies have
exactly 8 contigs** in `genome_final.parquet`.

The GTO bundles three things:
1. **Metadata** about the assembly (taxonomic ID, strain name, quality).
2. **Contigs** — the nucleotide sequences of each segment (the DNA layer).
3. **Features** — the structured annotations on those contigs (CDSs, mature peptides), each with a coordinate pointer back to its parent contig.

The DNA and annotation layers are held in two separate top-level
arrays (`contigs[]` and `features[]`); the only link between them is
the `contig_id` inside each feature's `location` field.

**Source-of-truth doc:** `docs/methods/gto_format_reference.md`
(complete schema + corpus statistics).
**Example file:** `docs/examples/example_gto_excerpt.json` (real
trimmed GTO from assembly `0000003012`, A/mallard/Wisconsin/394/1977
H4N1).

## §3. Top-level JSON structure

A flu GTO has these top-level keys (observed across all 200 sampled files):

```
analysis_events        # provenance: each annotation step + UUID + timestamp
close_genomes          # related-genome list (BV-BRC internal); not used
contigs                # ARRAY — one entry per nucleotide segment (DNA layer)
domain                 # 'Viruses'
features               # ARRAY — one entry per CDS or mat_peptide (annotation layer)
genetic_code           # 11 in 100.00% of flu GTOs in this corpus — see below
id                     # BV-BRC genome id (e.g. '197911.11517') — NOT used by Stage 1
ncbi_taxonomy_id       # e.g. 197911 = Influenza A virus
quality                # dict with 'genome_quality' field
scientific_name        # e.g. 'Influenza A virus (A/mallard/Wisconsin/394/1977(H4N1))'
```

**The `genetic_code = 11` artifact.** `genetic_code` is `11`
(bacterial / archaeal) in **1,793,563 / 1,793,563 (100.00%)** rows of
`protein_final.parquet`. Flu actually uses NCBI code 1; this is a
BV-BRC platform default, not a biological claim. Codes 1 and 11 share
the same start codon (ATG) and stop codons (TAA / TAG / TGA), so
nothing in this pipeline misbehaves — but downstream codon-level work
should hard-code code 1.

## §4. Contigs — the nucleotide layer (BIO + DATA)

Each entry in the `contigs` array is one **segment** of the assembly.
For flu, the typical (and complete) case is **8 contigs per GTO**.

| Field | Type | Meaning |
|---|---|---|
| `id` | str | BV-BRC contig identifier; this is the `contig_id` that `features[*].location` points back to. Format: `<genome_part>.<contig_index>` (e.g. `'1406633.10'`). |
| `dna` | str | The full nucleotide sequence of this contig, **5′ → 3′**, *including any 5′ and 3′ UTRs*. Uppercased before k-mer extraction. |
| `replicon_type` | str | Which flu segment this contig is — `'Segment 1'` ... `'Segment 8'`. |
| `replicon_geometry` | str | `'linear'` in 1,600 / 1,600 (100%) sampled flu contigs. |
| `contig_quality` | str | BV-BRC contig-level QC. `'Good'` in 108,530 / 108,530 (100%) of corpus contigs. |

**What is NOT on the contig:** no CDS coordinates, no UTR annotation,
no exon/intron annotation, no gene names. All of that lives in
`features[]`.

This matters for the k-mer pipeline: the nt features count k-mers over
**the full contig**, which includes UTRs and (for spliced segments)
intronic nucleotides. See §16.

## §5. Features — the annotation layer

Each entry in `features` is one annotation tied to (a portion of) a
contig. Only two feature types appear in flu GTOs:

- **`CDS`** — a complete coding sequence. 1,793,563 rows in
  `protein_final.parquet` (every entry).
- **`mat_peptide`** — a mature peptide cleaved out of a precursor.
  For flu, exclusively the HA1 and HA2 cleavage products of HA.
  Stage 1 explicitly **filters these out** of `protein_final.csv` and
  redirects them to `protein_non_cds.csv` (223,612 rows: 111,812 HA1 +
  111,800 HA2). The CDS-only filter is in `preprocess_flu.py:554-555`.

Per-feature fields (relevant subset):

| Field | Meaning |
|---|---|
| `id` | BV-BRC feature identifier; the canonical primary key. Parsed into `brc_fea_id`. Example: `'fig\|197911.84799.CDS.1'`. |
| `type` | `'CDS'` or `'mat_peptide'`. |
| `function` | Free-text protein function. **Join key against `conf/virus/flu.yaml`.** |
| `location` | Coordinate pointer back to contig(s). Single-entry for unspliced, multi-entry for spliced. See §6. |
| `protein_translation` | Amino-acid sequence of this feature. Parsed into `prot_seq`. Includes terminal `*` for the stop codon. |
| `family_assignments[0][1]` | Family ID (e.g. `Alphainfluenzavirus.PB2.1.pssm`). Parsed into `family`. |
| `feature_quality` | BV-BRC QC verdict. **Populated only on the 8 majors** (see §7). |

`feature_quality` is captured in 1,601 / 3,715 (43.1%) features in a
200-GTO sample — close to 200 × 8 = 1,600 (one major per segment per
isolate). It is captured but **never filtered on** by Stage 1.

## §6. The `location` schema — bridges features ↔ contigs

`location` is the load-bearing field for any nucleotide-level
interpretation. Schema:

```
location = [ [contig_id, start_1based, strand, length], ... ]
```

Four elements per inner span:
1. **`contig_id`** — points to the matching `contigs[i].id`.
2. **`start_1based`** — 1-based, inclusive start position on the plus strand. Position 1 = first nucleotide.
3. **`strand`** — `'+'` or `'-'`. **`'+'` in 100% of major-CDS rows in this corpus.** Parsers should handle `'-'` for portability.
4. **`length`** — span length in **nucleotides** (not an end coordinate). Inclusive end = `start + length - 1`.

### §6.1. Single-entry (unspliced) case

Example PB2 from the excerpt: `location = [["1406633.10", "16", "+", "2280"]]`.

```
contig "1406633.10"  (Segment 1, full length ≈ 2316 nt)

  pos:   1                16                                       2295         2316
         |                |                                          |            |
         v                v                                          v            v
  dna:  [5'-UTR (15 nt)]  [ATG · · · CDS (2280 nt = 760 codons) · · · TAG]  [3'-UTR]
                          └──────────────────── one span ────────────────┘

  Python slice:  contig.dna[16-1 : 16-1+2280]  =  contig.dna[15:2295]
```

- Span: nucleotides 16 ... 2295 on Segment 1. 2,280 nt / 3 = 760 codons.
- `prot_seq` is 760 chars long, ends with `*`, last 9 nt are `ATCAATTAG`.
- The identity `len(prot_seq) * 3 == location[0][3]` holds exactly — the annotated CDS nt length **includes the stop codon**, and the AA string includes the matching `*`.

### §6.2. Corpus checks on universal CDS properties

Verified against `data/processed/flu/July_2025/protein_final.parquet`:

- **CDS nt length is a multiple of 3:** 868,240 / 868,240 (**100.000%**) of major CDS rows.
- **Start codon is ATG:** 4,983 / 5,000 (**99.66%**) of sampled major CDSs. The 0.34% remainder is short non-canonical alternatives (GAA, ATA, AGA, CCA each ≤ 0.1%) — annotation artifacts.
- **Stop codon is one of TAA / TAG / TGA:** 4,951 / 5,000 (**99.02%**). Distribution: TGA 33.9%, TAA 33.5%, TAG 31.6%.

### §6.3. Multi-entry (spliced) case

**Splicing in flu.** Only **M2** (Segment 7) and **NEP** (formerly NS2, Segment 8) are spliced. All other flu proteins are unspliced.

In a GTO, splicing is encoded as multiple entries in the same `location` list, all pointing at the same `contig_id`. Each entry is one **exon**. Intron positions are *implicit* — whatever falls between two consecutive exons.

```
Segment 7 contig "1406633.6"   (full length ≈ 1027 nt)

  pos:    1     14                39 40                 727 728                995  1027
          |     |                  | |                    | |                    |    |
          v     v                  v v                    v v                    v    v

  M1 :   [utr]  [ATG─── M1 CDS (singly continuous, 759 nt) ───────────────TAA] [utr]
                ▲                                                            ▲
                └─── shared start codon ───┐                                 └─ M1 stop codon
                                           │
  M2 :   [utr]  [ATG─exon1 (26 nt)─]  ╳╳╳╳╳╳ intron (688 nt) ╳╳╳╳╳╳ [─exon2 (268 nt)─TAA] [utr]
                └────────── M2 mature CDS = exon1 + exon2 = 26 + 268 = 294 nt ─────────┘

  M2 location:
    [["1406633.6", "14",  "+",  26],   ← exon 1: nt 14 ... 39
     ["1406633.6", "728", "+", 268]]   ← exon 2: nt 728 ... 995
```

M1 and M2 share the start codon (both begin at position 14 on Segment 7) but use different reading frames after the splice donor — M2 reads through a frame shift introduced by removing the intron.

**Importantly:** for the spliced features (M2, NEP), the **`prot_seq` stored in `protein_final` is the spliced protein** — but the **`dna_seq` we later attach (from `genome_final`) is the full contig**, intron included. The k-mer pipeline counts k-mers over the contig, *not* the spliced mature mRNA. See §16.

## §7. Major proteins and the segment ↔ protein map

`conf/virus/flu.yaml` distinguishes:

- **`core_functions`** (9 entries) — proteins present in nearly all Flu A genomes (PB2, PB1, PA, HA, NP, NA, M1, M2, NEP).
- **`aux_functions`** (11 entries) — additional products that may or may not be present.
- **`selected_functions`** (8 entries) — the **major proteins** used for ML modeling. **One primary product per segment.**

The 8 majors: PB2, PB1, PA, HA, NP, NA, M1, NS1. (M2 and NEP are core but not major — they share segments 7/8 with the major and use alternative reading frames.)

|---|---|---|---|---|

when **both** its `function` and its parent contig's `replicon_type`
match. If a record says "PB2 on Segment 3", that's inconsistent and
the protein is left unassigned and dropped at Stage 1 filter step 1.

## §8. Coding vs non-coding nucleotides (BIO + DATA)

GTOs annotate **only coding regions** explicitly. UTRs and introns
exist on the genome but have **no `features[i]` entry** of their own —
they are implicit, computable as the complement of CDS coverage on a
contig.

# Part B — Preprocessing (Stage 1)

## §10. What Stage 1 does and emits (DATA)

**Script:** `src/preprocess/preprocess_flu.py`.
**Inputs:** `*.gto` files in `data/raw/<run_dir>/`.
**Outputs:**
- `data/processed/flu/{ver}/protein_final.{csv,parquet}` — one row per protein feature, **1,793,563 rows**.
- `data/processed/flu/{ver}/genome_final.{csv,parquet}` — one row per contig, **868,240 rows** (= 108,530 isolates × 8 segments).
- `protein_non_cds.csv` — the redirected `mat_peptide` rows (223,612: 111,812 HA1, 111,800 HA2).
- A handful of QC reports (`prot_df_no_seq`, `genome_missing_seqs.csv`, etc.).

**Per-GTO flow** (`preprocess_flu.py:61` is the only place GTO JSON is read):
1. Parse JSON once. Build a per-isolate `meta_df`, a `segment_map` (contig id → replicon_type), and two tables — one row per feature, one row per contig.
2. Aggregate across all GTOs into `prot_df` and `genome_df`.
3. Run the protein pipeline (segment assignment → basic filters → sequence-level dedup → ESM-2 prep). See §12.
4. Run the genome pipeline (segment assignment → basic filters → DNA QC → DNA dedup). See §13.
5. Emit `protein_final.{csv,parquet}` and `genome_final.{csv,parquet}`.

**Stage 3 join key:** `(assembly_id, genbank_ctg_id)` is used by Stage 3
(`_pair_helpers.py::attach_dna_to_prot_df`) to attach DNA to protein rows.

## §11. GTO field map and silent drops

| GTO field | Where used in `protein_final` / `genome_final` |
|---|---|
| `ncbi_taxonomy_id`, `genetic_code`, `scientific_name` | meta → protein rows. KeyError on absence (fail-loud). NOT joined into genome rows (asymmetry, see §13). |
| `quality.genome_quality` | both tables, as column `quality`. Drives the "Poor" filter. Single value per assembly (not per segment — see issue #5 in §15). |
| `contigs[*].{id, replicon_type, contig_quality, dna}` | genome rows. `replicon_type` defaults to `'Unassigned'` if absent. `contig_quality` is **captured but never filtered on** (issue #6). |
| `features[*].{id, type, function, protein_translation, location, feature_quality, family_assignments}` | protein rows. `feature_quality` populated only on the 8 majors. |
| `location[0][0]` | derives `genbank_ctg_id`. For spliced features (M2, NEP, PA-X, NS3, M42, PB2-S1) both entries always point at the same contig in this corpus — verified 100% — so `[0][0]` is safe. |
| `family_assignments[0][1]` | column `family` (e.g. `Alphainfluenzavirus.PB2.1.pssm`). |

**Silently dropped top-level fields:** `domain`, `analysis_events`,
`close_genomes`, and **`gto['id']`** (the BV-BRC genome ID). The
pipeline derives `assembly_id` from the **filename** instead
(`00621bf88c.gto` → `00621bf88c`). Different ID spaces; happen to be
1:1 per file. The BV-BRC genome ID can be recovered from `brc_fea_id`
(`fig|197911.11517.CDS.N`).

**Sequence hashes (derived).**
- `seq_hash = md5(prot_seq)` — protein-level identity. Written by Stage 1.
- `dna_hash = md5(dna_seq)` — DNA-level identity. Added by Stage 3 (`attach_dna_to_prot_df`).

The unprefixed `seq_hash` is the *protein* hash for historical reasons;
DNA gets the explicit `dna_` prefix. In pair tables these become
`seq_hash_a` / `seq_hash_b` and `dna_hash_a` / `dna_hash_b`.

The ESM-2 cache key uses `sha1(prot_seq)` (different namespace, never
joined back).

## §12. Protein-side filter pipeline (DATA)

`apply_protein_basic_filters` → `handle_protein_duplicates` →
`prepare_sequences_for_esm2`. In order:

1. Drop rows where `assign_protein_segments` left `canonical_segment` null. (Function + replicon_type didn't match a known biological mapping.)
2. Keep `type == 'CDS'` only. **This is where `mat_peptide` rows (HA1, HA2) are redirected to `protein_non_cds.csv`.**
3. Drop `quality == 'Poor'` (genome-level — see §14 caveat).
4. Drop `replicon_type == 'Unassigned'`.
5. `handle_assembly_duplicates`: collapse `(prot_seq, assembly_id)` duplicates. For Flu A every file is one assembly, so the GCF/GCA preference branch (`gto_utils.py:181-185`) is effectively dead code; "same file → keep first" handles all real cases.
6. `prepare_sequences_for_esm2`: drop sequences exceeding internal-stop or X-fraction thresholds, impute `X → G`, strip terminal `*` from `prot_seq`.
7. Final cut: `prot_df = prot_df[prot_df['esm2_ready_seq'].notna()]`.

**Segment assignment is biology-validated.**
`assign_protein_segments` (`preprocess_flu.py:481-537`) joins on
**both** `function` and `replicon_type` against
`conditional_segment_mappings` in `conf/virus/flu.yaml`. A GTO record
claiming "PB2 on Segment 3" stays unassigned and is dropped at step 1.
This is a strong invariant: a row in `protein_final.csv` is
guaranteed biology-consistent at the `(function, replicon_type)` level.

## §13. Genome-side filter pipeline

`apply_genome_basic_filters` → `handle_genome_duplicates`:

1. Drop null `canonical_segment`.
2. Drop `quality == 'Poor'` (same genome-level limitation).
3. Drop `replicon_type == 'Unassigned'`.
4. Drop missing `dna_seq`.
5. `summarize_dna_qc`: annotate `length`, `gc_content`, `ambig_count`, `ambig_frac`.
6. Drop `(dna_seq, assembly_id)` duplicates (keep first).

**Asymmetry with protein side.** Genome rows do NOT carry
`assembly_prefix, ncbi_taxonomy_id, genetic_code, scientific_name`.
Stage 3's join only pulls `dna_seq`, so this isn't a correctness
issue — but a genome row alone can't answer "what taxonomy / assembly
prefix is this?". Cheap fix exists; tracked as issue #4.

## §14. What is and isn't quality-filtered (BIO + DATA)

This answers question (1b) — **how do we filter sequences for quality?**

| Layer | Filter applied | Filter NOT applied |
|---|---|---|
| Genome | `quality == 'Poor'` (genome_quality field) | `contig_quality` is captured but **never used** (issue #6). |
| Protein | CDS-only; `canonical_segment` null drop; `(prot_seq, assembly_id)` dedup | `feature_quality` is captured but **never used** (issue #6). |
| ESM-2 prep | `max_internal_stops = 0.1` (drop), `max_x_residues = 0.1` (drop), X→G imputation, `strip_terminal_stop = True` | These thresholds are **hardcoded** in `preprocess_flu.py:1159-1162`, not in `conf/virus/flu.yaml`. Issue #2. |
| DNA QC | `summarize_dna_qc` annotates `ambig_frac`; rows are not filtered on it. | Pathological ambiguity propagates to Stage 2b k-mer counting, which then *skips* affected windows (§18). |

So the answer to "do we filter for quality" is **yes for genome-level Poor
and CDS-type; partial for protein sequence pathology (ESM-2 prep); no for
per-contig contig_quality**.

**Length semantics in outputs:**
- `protein_final.length` = **AA length as parsed from the GTO**, including the trailing `*`. Captured at parse time and never recomputed. After ESM-2 prep, the terminal `*` is stripped, so `len(esm2_ready_seq) == length - 1` in 1,771,244 / 1,793,563 (98.76%) rows and `len(esm2_ready_seq) == length` in 22,319 (1.24%) — the 1.24% are entries without a terminal stop. For ML-input length, use `len(esm2_ready_seq)`.
- `genome_final.length` = nucleotide length.

# Part C — K-mer feature computation (Stage 2b)

## §16. Source sequences — what nt and aa actually come from (DATA + BIO)

This answers question (2a) directly.

**`kmer.alphabet = nt`** (current production for `flu_*_kmer_nt_*` and `flu_ha_na`, `flu_pb2_pb1`):
- Source: `genome_final.csv → dna_seq` — **one row per contig**, the FULL contig DNA (5′ UTR + CDS + 3′ UTR; for spliced segments S7/S8 the full contig **includes intronic nucleotides**).
- **Not CDS-scoped.** A 6-mer counted in a flu nt cache may sit inside the CDS, inside a UTR, or straddle a CDS/UTR boundary. For spliced segments it may sit inside an intron.
- Index key: `(assembly_id, genbank_ctg_id)`.

**`kmer.alphabet = aa`** (current production for `flu_*_kmer_aa_*`):
- Source: `protein_final.csv → prot_seq` — **one row per protein feature**, the full translated protein sequence including (for spliced features) the splice-joined product. **Not contig-scoped.**
- Index key: `(assembly_id, brc_fea_id)`.

**Why not CDS-scoped on the nt side?** Stage 2b reads `genome_final`,
which is one row per contig. There is no per-CDS DNA column. Reconstructing CDS DNA would require slicing the contig with the `location` field — a one-page helper (§9 of `gto_format_reference.md`
gives the recipe) that nothing in the codebase performs today.

**Why this matters for interpretability.** Stride-1 k-mer features mix
all three reading frames + UTRs + introns. Feature importances cannot
be read as codon-level signal. The plan
`docs/plans/2026-05-12_codon_aware_kmer_features_plan.md` is the
follow-up that would change this.

## §17. The counting algorithm

```python
def compute_kmer_counts(seq: str, k: int, alphabet: str = 'ACGT') -> Counter:
    counts = Counter()
    seq_upper = seq.upper()
    valid = set(alphabet)
    for i in range(len(seq_upper) - k + 1):
        kmer = seq_upper[i:i + k]
        if all(c in valid for c in kmer):
            counts[kmer] += 1
    return counts
```

**Parameters (production):**
- **k** — nt: `6`; aa: `3`. (Bundle-specific; both nt-k3 and nt-k6 bundles exist.)
- **stride** — `1`. Hard-coded in the loop above; not a CLI parameter.
- **alphabet** — `'ACGT'` for nt, `'ACDEFGHIKLMNPQRSTVWY'` (canonical 20-AA, no B/X/Z/J) for aa.
- **normalize** — `none` in production. Length normalization (`l1`) and magnitude (`l2`) are supported but not used in paper experiments. The MLP + slot transform handles raw counts.

**No reverse-complement canonicalization.** The counter operates on
the DNA string as stored in the GTO (sense strand for each contig).
We do not sum counts of each k-mer with its reverse complement.
Possible ablation; not exercised today.

**Position-agnostic.** The representation is a multiset. Two sequences with
the same 6-mer composition but different arrangements produce identical
feature vectors.

## §18. Ambiguity handling (answers 2b)

Windows containing **any** non-`ACGT` (or non-canonical 20-AA) character
are **silently skipped, not imputed**. For flu the impact is
negligible — the `ambig_frac` column in `genome_final.csv` is near 0
for well-assembled sequences. A segment with pathological ambiguity
would produce a partial k-mer profile; Stage 1 QC (`genome_missing_seqs.csv`,
`genome_poor_quality.csv`) drops the worst offenders before Stage 2b sees them.

**Contrast with ESM-2 prep:** Stage 1's `prepare_sequences_for_esm2`
imputes `X → G` and strips terminal `*` for protein sequences fed to
ESM-2. Stage 2b's aa path uses `prot_seq` directly (not
`esm2_ready_seq`), and **skips** any window containing a non-canonical
AA — so the aa cache is naturally tolerant of ambiguity without imputation.

## §19. Final dimensions and storage layout (answers 2c)

Verified against on-disk artifacts 2026-05-13:

| Cache | Matrix rows | Vocabulary | Index rows (occurrences) | NPZ size |
|---|---:|---:|---:|---:|
| `kmer_features_nt_k6.npz` | 868,240 | 4,096 (4⁶) | 868,240 | 1.78 GB |
| `kmer_features_nt_k3.npz` | 868,240 | 64 (4³) | 868,240 | 25 MB |
| `kmer_features_aa_k3.npz` | 375,413 (unique `prot_seq`) | 8,000 (20³) | 1,793,563 | 71 MB |

**Storage format:** `scipy.sparse.csr_matrix` saved as
`kmer_features_{nt|aa}_k{K}.npz`. Companion `*.parquet` index maps
composite occurrence keys to matrix rows; `*.json` carries metadata
(k, alphabet, normalize, build timestamp, etc.). Cells are `float32`
because common k-mers in long sequences can exceed int8's 127 cap, and
because normalization paths require fractional values.

**The aa cache is sequence-deduplicated** on write: 1,793,563
protein-occurrence rows collapse to 375,413 unique `prot_seq` (20.9%
retention), keyed internally by `md5(prot_seq)` — same pattern as the
ESM-2 cache. The parquet index records the N-to-1 mapping. The nt
cache currently stores **one row per contig occurrence** (no dedup);
Phase 6 of `docs/plans/2026-05-13_aa_kmer_and_cache_symmetry_plan.md`
is the planned migration that will dedup nt by `md5(dna_seq)`.

**Why the matrix has 868,240 rows.** 108,530 isolates × 8 segments = 868,240. Multiple proteins can share one segment (M1/M2 on S7, NS1/NEP on S8, PA/PA-X on S3, PB1/PB1-F2/PB1-N40 on S2, PB2/PB2-S1 on S1). Those proteins receive **identical** k-mer vectors when looked up — semantically correct (they share contig DNA). The 28-pair sweep bundles select exactly one function per segment to avoid this.

## §20. Pair-feature construction and interactions

For a pair generated by Stage 3:
1. `row_a = index_lookup(assembly_id_a, ctg_a)` (nt) or `(assembly_id_a, brc_a)` (aa)
2. `row_b = index_lookup(assembly_id_b, ctg_b)` / `brc_b`
3. `v_a = kmer_matrix[row_a].todense()`
4. `v_b = kmer_matrix[row_b].todense()`
5. `pair_feature = f(v_a, v_b)`

**Supported interactions `f(·, ·)`** (see §34 for the modeling-side discussion):

| Token | Formula | Output dim |
|---|---|---:|
| `concat` | `[v_a ; v_b]` | 2× |
| `diff` | `\|v_a − v_b\|` (element-wise abs) | 1× |
| `unit_diff` | `\|v_a − v_b\| / ‖ \|v_a − v_b\| ‖₂` (abs first, then L2 — symmetric) | 1× |
| `prod` | `v_a ⊙ v_b` (element-wise) | 1× |
| `unit_prod` | `(v_a ⊙ v_b) / ‖v_a ⊙ v_b‖₂` | 1× |

Tokens compose with `+`, e.g. `unit_diff + prod` produces an
`[unit_diff(v_a,v_b) ; prod(v_a,v_b)]` 2× vector. **Current production
HA/NA and PB2/PB1 k-mer bundles use `unit_diff + prod` with
`slot_transform: unit_norm`.**

**Per-slot transform** applied before the interaction:
- `none` — raw.
- `unit_norm` — parameter-free L2 row-norm. Supported in both the MLP and the k-NN baseline path (`get_kmer_pair_features`).
- `slot_norm` — learnable LayerNorm. **Not allowed on k-mer features** (raises a ValueError at `get_kmer_pair_features:113-119`) because non-negative count vectors don't benefit from it.

## §21. Scaling and practical limits (ML + DATA)

The current pipeline uses **exhaustive enumeration of the
`|alphabet|^k` vocabulary**. This is fine for small k but hits walls
quickly on aa. Three independent bottlenecks:

### Bottleneck 1 — vocabulary enumeration (Python memory at build time)

| alphabet | k | Vocab size | Build memory |
|---|---:|---:|---:|
| nt (4)  | 6 | 4,096 | trivial |
| nt      | 10 | 1,048,576 | ~200 MB |
| aa (20) | 3 | 8,000 | trivial |
| aa      | 4 | 160,000 | ~30 MB |
| aa      | 5 | 3,200,000 | ~700 MB |
| aa      | 6 | 64,000,000 | ~10 GB |
| aa      | 7 | 1,280,000,000 | OOM on any reasonable machine |

### Bottleneck 2 — sparse cache size on disk

NNZ per sequence is bounded by `(L − k + 1)` distinct k-mers, where
`L` ≈ 500 aa for a typical flu protein. Total NNZ ≈ `N_seqs × ~498`
once `k ≥ 3`. CSR is ~70 MB at aa k=3 and stays ~1–2 GB at any larger
k. **Disk is not the bottleneck.**

### Bottleneck 3 — MLP input dim (the actual wall at training time)

`EMBED_DIM = |alphabet|^k`. With default `hidden_dims[0] = 512`:

| alphabet | k | EMBED_DIM | First Linear layer params | Weights (fp32) |
|---|---:|---:|---:|---:|
| nt | 6 | 4,096 | 2.1M | 8 MB |
| nt | 10 | 1,048,576 | 537M | 2.1 GB |
| aa | 3 | 8,000 | 4.1M | 16 MB |
| aa | 4 | 160,000 | 82M | 328 MB |
| aa | 5 | 3,200,000 | 1.6B | 6.5 GB |
| aa | 6 | 64,000,000 | 32B | 128 GB |

Densification at training time multiplies this further:
`KmerPairDataset` does `kmer_matrix[unique_rows].todense()`, so a
single dense row at aa k=6 is 256 MB and a batch of 128 pairs × 2
slots is ~64 GB — infeasible even before the MLP.

### Bottom line

- **nt** practical up to k≈10; production is k=6 (comfortable).
- **aa** practical up to k=4; production is k=3.
- **aa k≥5** needs a redesign: observed-vocab (only enumerate k-mers actually seen) or feature hashing (fixed-size index space). Neither is implemented.

## §26. Phase 3 — Split routing (seq_disjoint, hash_key=seq)

With `split_strategy.mode = seq_disjoint` and `hash_key = seq`, the post-dedup positive pairs are routed to train/val/test such that **no `seq_hash` appears in two splits** on either slot.

**Algorithm** (`_pair_helpers.seq_disjoint_route_pos_df`):
1. Build a bipartite graph with slot-A `seq_hash`es as one side, slot-B `seq_hash`es as the other, and an edge per positive pair.
2. Find connected components (each CC is a set of pairs that share at least one `seq_hash` on either side, transitively).
3. Bin-pack components into 3 buckets sized to the train/val/test ratios using **LPT-greedy** (largest-first).
4. Hard-fail if the resulting routing produces nonzero overlap on the active hash family.

`hash_key=seq` is **strictly stronger than `hash_key=dna`**: the
aa-share set ⊇ dna-share set, because identical DNA implies identical
AA (so it would be in the dna-share set), and there are *more*
aa-shared pairs from synonymous-codon variation that wouldn't show up
under dna keying. Routing on `seq` therefore constrains more pairs and
provides BOTH aa AND dna leakage prevention at construction time.

|---|---:|---:|---:|---:|---:|---:|---:|
| `flu_ha_na*` | 21,719 | 11,748 | 20.1% | 17,724 | 41,629 | 37,207 | **0** |
| `flu_pb2_pb1*` | 14,924 | 20,214 | **38.4%** | 12,384 | 33,414 | 30,996 | **0** |

**Targets vs achieved** (integer-pair):

`max_target_deviation_pct = 0.0007%` (HA/NA) and `0.0011%` (PB2/PB1) — both are integer-rounding artifacts (the fractional target is `0.8 × n_pairs`, which is never a clean integer).

**The PB2/PB1 largest CC at 38.4%** is the conservation effect: PB2/PB1 has fewer distinct proteins overall, so the bipartite graph has a more concentrated connected component. It still fits inside the train bucket, so 80/10/10 is hit cleanly.

## §27. Phases 4–5 — Negative sampling (coverage, then regime-aware fill)

Negative pairs are sampled across isolates with three guarantees:

1. **Co-occurrence blocking.** A candidate `(seq_hash_a, seq_hash_b)` is rejected if it appears in `total_cooccur_pairs` (any positive, anywhere). Counted in `co_occurrence_blocking.{train,val,test}_blocked`. Without this, "negatives" could secretly be positive pairs that landed in a different isolate.
2. **Coverage.** Every sequence (per slot, per `seq_hash` and per `dna_hash`) that appears in positives must appear in ≥1 negative in the same split.
3. **Regime mix** (opt-in, Phase 5 only).

### Phase 4 — Coverage phase (regime-blind)

Walks every `(slot, dna_hash)` target and finds at least one valid negative partner. This is the protein-level imbalance fix (mode #2 leakage) and the DNA-level best-effort variant. A hard-fail invariant: **`n_seqs_with_zero_negatives = 0`** at the protein level (raises at `dataset_segment_pairs_v2.py:819` otherwise).

### Phase 5 — Regime-aware fill phase

Only runs when `negative_sampling.regime_targets` is set (the
`*_regimes` bundles). Tops up to `num_negatives = round(neg_to_pos_ratio × |pos|)` while biasing toward the configured mix.

**The 8 regimes** (per `_negative_regime_sampling.REGIME_NAMES`):

| Regime | Match condition | Hardness |
|---|---|---|
| `none_match` | host ≠ AND subtype ≠ AND year ≠ | easiest |
| `host_only` / `subtype_only` / `year_only` | exactly one axis matches | one shortcut |

# Part E — Leakage taxonomy and what's mitigated

## §29. Five canonical leakage modes (ML — slide-able overview)

Canonical names from `docs/methods/leakage_definitions.md`. **Use these names verbatim; don't invent synonyms.**

| # | Canonical name | Description |
|---|---|---|
| 1 | **Same-pair leakage** | Same `pair_key` appears in train and test. |
| 2 | **Sequence-level label imbalance** | A sequence appears only as positive (or only as negative) within a split. |
| 3 | **Sequence-level leakage** | Same `seq_hash` / `dna_hash` appears in different pairs across splits. |
| 4 | **Cluster leakage** | Test pair's feature vector is cosine-near a training pair's, even if no exact hash match. |
| 5 | **Demographic shortcut leakage** | Model uses `same_host` / `same_subtype` / `same_year` etc. as proxy for "same isolate." |

Disambiguation:
- **#2 vs #3** — both touch sequence identity. #2 is about how *labels* of a single sequence are distributed (only-positive or only-negative is bad). #3 is about whether the *same identifier* appears in two splits.
- **#3 vs #4** — both touch cross-split similarity. #3 is exact match (same hash). #4 is approximate match in feature space (high cosine).
- **#5 vs #1–#4** — modes 1–4 are data-level contamination. Mode 5 is the model preferring metadata correlations to sequence content — a shortcut, not contamination.

## §30. What the current pipeline mitigates by construction

| # | Mode | Status in this pipeline |
|---|---|---|
| 1 | Same-pair leakage | ✅ Mitigated — v2 cross-split `pair_key` assertion raises on any overlap (audit at save time). |
| 2 | Sequence-level label imbalance | ✅ Mitigated — v2 coverage phase iterates `(slot, seq_hash)` and `(slot, dna_hash)`. `n_seqs_with_zero_negatives = 0` and `n_dna_uncovered = 0` on every in-scope bundle (§27). |
| 3 | Sequence-level leakage | ✅ Mitigated by construction — `seq_disjoint` routing with `hash_key=seq`. Cross-split overlap = 0 on the active family. Zero pair drops on all four bundles (§26). |
| 4 | Cluster leakage | ⚠️ Partially. Exp 2 (1-NN baseline `knn1_margin`) and Exp 3 (cosine deciles) implemented. Exp 4 (`seq_disjoint`) bounds the exact-hash case. **Exp 5 (mmseqs2 cluster splits) not yet implemented.** PB2/PB1 finding: 1-NN ≈ MLP on MCC (0.900 vs 0.887), inside seed noise — see §32. |
| 5 | Demographic shortcut | ⚠️ Confirmed present in legacy random-sampled datasets (30–50× FP rate climb from `none_match` to `host_subtype_year`). Construction-time mitigation: regime-aware sampling (`*_regimes` bundles). **Re-test on regime-built datasets is the open assessment** — done via the per-regime heatmap (§9). |

**Mode #5 cannot be fully fixed by sampling alone.** Even with a
balanced regime mix, the model can still pick up the metadata pattern
in training. The per-regime heatmap is the test that says whether the
mitigation worked.

## §31. The aa-vs-nt similarity-leakage diagnostic (new, 2026-05-13)

This is a mode #4 (cluster leakage) diagnostic at the **slot** level: for each unique test sequence, find the max % identity over all unique train sequences. Identity-disjointness (no shared `seq_hash`) is enforced by `seq_disjoint` — but a test protein 1 aa away from a train protein is still **similar**, even if not identical.

**Hypothesis:** because synonymous codon variation creates DNA diversity that aa is invariant to, the **aa representation should see closer nearest-train neighbors than the dna representation.**

**Method:** for each unique test sequence, compute `matches / max(L_train, L_test) × 100` after right-padding with distinct pad bytes (train=0, test=255) so pad-pad positions can't spuriously match.

**Results on the HA/NA regimes dataset** (script: `src/analysis/similarity_leakage_aa_vs_nt.py`; doc: `docs/results/2026-05-13_aa_vs_nt_similarity_leakage.md`):

| Side / level | Train unique | Test unique | Mean % | Max % | % ≥ 99.5% | % < 95% |
|---|---:|---:|---:|---:|---:|---:|
| HA **aa** | 29,951 | 5,839 | 98.91 | 99.82 | **47.83** | 1.99 |
| HA **dna** | 42,247 | 5,839 | 97.97 | 99.94 | 41.58 | 6.79 |
| NA **aa** | 25,529 | 5,839 | 98.98 | 99.79 | **55.06** | 2.02 |
| NA **dna** | 39,417 | 5,839 | 98.35 | 99.93 | 44.89 | 5.20 |

**Findings:**
1. Splits are identity-disjoint as designed — max identity capped just below 100% in every row (HA aa max = 99.82% = one aa change in 567).
2. **AA-level similarity leakage exceeds DNA-level by 6–10 pp at ≥99.5%.** Mean nearest-train identity is 0.6–0.9 pp higher at aa. Matches biological prediction: synonymous variation makes the dna representation more diverse.
3. DNA has a longer left tail (3× more cases below 95% identity), because identical-AA proteins can have substantially different DNA encodings.

**Implication for nt-vs-aa k-mer comparison:** in the 2026-05-13 HA/NA training comparison, aa k=3 + unit_diff+prod achieved F1=0.925 vs nt k=6 + unit_diff+prod F1=0.911 (Δ=−0.014). The similarity-leakage asymmetry (+0.6 to +0.9 pp mean, +6 to +10 pp on high-similarity bucket) is in the same order as the F1 gap. **Cannot conclude aa's lead is due to representation power alone — it could be similarity leakage.** Clean test: mmseqs2 cluster-disjoint splits (plan §5 of `docs/plans/2026-05-08_cosine_and_cluster_splits_plan.md`).

## §32. The "biology learning" criterion (ML)

The model has learned biology-relevant signal **insofar as** its accuracy on (a) and (b) is meaningfully higher than a 1-NN lookup on the same splits and features:

(a) **Generalization to novel sequences** — pairs whose individual sequences have cosine < 0.95 to all training sequences, or fall in a test cluster disjoint from any training cluster (Plan Exp 5, mmseqs2 at 95% identity).

(b) **Generalization across populations** — pairs from a held-out demographic stratum (subtype, host, year) that does not appear in training.

If MLP ≈ 1-NN on these splits, the model is doing nothing more than near-neighbor lookup. **Initial bar: MLP must beat 1-NN by > 0.02 AUC.**

**Status (2026-05-12):**
- **HA/NA** under `hash_key=dna` routing (Exp 4a, 2026-05-11): MLP `host_subtype_year` TNR drops 0.872 → 0.834. MLP still leads 1-NN on aggregate metrics. ✅ pass.
- **PB2/PB1** under `hash_key=seq` routing (2026-05-12): 1-NN edges MLP on aggregate MCC (0.900 vs 0.887). **Inside seed noise.** ⚠️ inconclusive — we cannot currently claim the MLP has learned biology beyond near-neighbor lookup on PB2/PB1.

This is the headline result the experiment matrix in §37 will sharpen, especially with the regime-aware bundles where the per-regime heatmap directly tests mode #5.

# Part F — Modeling

## §33. Slot transforms (answers 4a part 1)

Applied to `v_a` and `v_b` independently before the pair interaction:

| Token | Op | Use in production |
|---|---|---|
| `none` | identity | Default for LightGBM (tree splits are scale-invariant). |
| `slot_norm` | learnable LayerNorm per slot | **Critical for ESM-2** to cancel the protein-type subspace offset (HA / NA embeddings live in slightly different subspaces in ESM-2 space; LayerNorm removes the offset). Used in older gen-3 bundles. Raises on k-mer features (non-negative counts don't benefit from it). |
| `unit_norm` | parameter-free L2 row-norm | Current production for HA/NA & PB2/PB1 k-mer bundles. Supported in both the MLP and the k-NN baseline path (`get_kmer_pair_features`). |

**Why `unit_norm` for k-mer:** k-mer count vectors are non-negative and carry a length confound (longer contigs accumulate more counts). L2 row-norm removes magnitude as a free variable while preserving directional information.

## §34. Pair interactions in detail (answers 4a part 2)

| Token | Formula | Output dim (k=4096 each side) | Semantic |
|---|---|---:|---|
| `concat` | `[v_a ; v_b]` | 8,192 | Both vectors, in order. Captures absolute identity of each slot. |
| `diff` | `\|v_a − v_b\|` (abs) | 4,096 | Element-wise magnitude of difference. Symmetric in a, b. |
| `unit_diff` | `\|v_a − v_b\| / ‖ \|v_a − v_b\| ‖₂` | 4,096 | Direction of difference. Symmetric since 2026-05-12 (abs first, then L2). The previous *signed* normalization is gone. |
| `prod` | `v_a ⊙ v_b` | 4,096 | Element-wise product. Symmetric. Picks out k-mers present in both. |
| `unit_prod` | `(v_a ⊙ v_b) / ‖v_a ⊙ v_b‖₂` | 4,096 | L2-normalized prod. |

Tokens compose with `+`: `unit_diff + prod` = `[unit_diff(v_a, v_b) ; prod(v_a, v_b)]` (8,192-dim for k-mer). **Current production for k-mer HA/NA and PB2/PB1.**

### Why ESM-2 needs `unit_diff` over `concat` (key empirical finding)

- **ESM-2 `concat` collapses on H3N2-only HA/NA**: AUC = 0.50. **ESM-2 `unit_diff` succeeds**: AUC = 0.96.
- Apr 2026 all-pairs sweeps (Task 11): ESM-2 `concat` collapses on PB1/PA on BOTH the full unfiltered dataset (AUC=0.4971) AND the H3N2-filtered dataset (AUC=0.4981). The subspace-offset failure is a property of **ESM-2's protein-type geometry**, not of a specific subtype.
- Direction of embedding difference carries genuine biological signal; magnitude is a shortcut.

### Why k-mer doesn't have this problem

- **K-mer `concat` does NOT collapse on H3N2**: AUC = 0.985.
- K-mer features are interaction-agnostic (unit_diff ≈ concat ≈ prod) because sparse frequency vectors don't have ESM-2's subspace offset.
- On H3N2-only: k-mer unit_diff AUC=0.988 vs ESM-2 unit_diff AUC=0.957.

## §35. MLP architecture and training

Per-slot transform → pair interaction → 3-layer MLP → BCE loss.

**Current production hyperparameters** (from `conf/training/base.yaml` + bundle overrides):
- Hidden dims: `[512, 256]` (3-layer net = 2 hidden + 1 head).
- Activation: ReLU.
- Dropout: 0.2.
- Optimizer: Adam, lr 1e-3, weight_decay 1e-5.
- Batch size: 256.
- Early stopping: monitor val loss, patience 10 (40+ for H3N2-only).
- Max epochs: 100.
- Master seed: 42 (hierarchical seed system, see `docs/SEED_SYSTEM.md`).

**Metrics logged to `metrics.csv`:** F1 (binary + macro), precision, recall, AUC-ROC, AUC-PR (avg precision), MCC, Brier, BCE loss. Early-stop options: `loss`, `f1`, `auc` (=ROC), `auc_pr`, `mcc`.

### Key empirical findings to date

- **`unit_diff` > `concat` on homogeneous data (ESM-2 only)** — covered above.
- **LayerNorm (`slot_norm`) is critical for ESM-2 on homogeneous subsets**: without it, raw HA/NA embeddings live in slightly different subspaces; `unit_diff` then picks up slot offset rather than biological signal.
- **Delayed learning on H3N2 + unit_diff**: characteristic plateau-then-breakthrough (epochs ~10–32, seed-dependent). Increase `patience` to 40+ for H3N2-only runs.
- **K-mer (k=6, 4096-dim) matches or exceeds ESM-2 on mixed-subtype HA/NA**: AUC 0.982 vs 0.966–0.975.
- **High FP rate on filtered datasets**: likely subtype/host confounders in negative pairs — model learns "same population" rather than "same isolate." Mode #5 (demographic shortcut); the regime-aware sampler is the construction-time fix; the heatmap is the assessment.
- **PB2/PB1 seq_disjoint: 1-NN baseline edges MLP** (MCC 0.900 vs 0.887). Consistent with conservation effect.

## §36. Baselines

All five models train on the same Stage 3 dataset and produce the same `test_predicted.csv` schema. Each writes its own `models/flu/{ver}/runs/training_<bundle>_<TS>/` directory.

| Model | Role |
|---|---|
| **MLP** | The production classifier. |
| **Logistic regression** | Linear baseline. Preprocessed with `StandardScaler` (per-feature column-wise — linear models need scale parity). Tells us how much signal is captured by an affine combination of features. |
| **LightGBM** | Tree-based baseline. No preprocessing (tree splits are scale-invariant). Captures non-linear interactions without representation learning. **Strong default baseline for k-mer features.** |
| **1-NN cosine-margin (`knn1_margin`)** | The **leakage anchor.** Score = cosine to nearest train positive minus cosine to nearest train negative. If MLP ≈ 1-NN on AUC, the MLP is doing soft near-neighbor lookup, not generalization. |
| **k-NN vote (`knn_vote`)** | Smoothed neighborhood baseline. k=5 by default, distance-weighted. |

**Caveat for the heatmap.** Per-slot transform interacts differently with each model. Under current k-mer production (`unit_norm`), MLP and k-NN baselines see identical slot-level preprocessing. LR uses its own `StandardScaler` (column-wise). LightGBM uses none. The heatmap caption flags this so reviewers don't read row-to-row differences as pure model-quality differences.
