## Project Summary
Flu A viral segment co-occurrence prediction. ESM-2 protein embeddings (frozen) + MLP binary classifier.
Primary virus: Influenza A. Bunya support exists but NOT actively maintained.

## Pipeline (4 stages)
- Stage 1: `src/preprocess/preprocess_flu.py` → `data/processed/flu/{version}/protein_final.csv` + `genome_final.csv` (run once). Invoked by `scripts/stage1_preprocess_flu.sh`. Unified protein + genome extraction in one pass. (The legacy `preprocess_flu_protein.py` was deleted 2026-05-13 after the unified script absorbed its logic; recoverable via git history if ever needed.)
- Stage 2: `src/embeddings/compute_esm2_embeddings.py` → `data/embeddings/flu/{version}/master_esm2_embeddings.h5` (run once)
- Stage 3: `src/datasets/dataset_segment_pairs.py` → `data/datasets/flu/{version}/runs/dataset_{bundle}_{ts}/` (per experiment)
- Stage 4: `src/models/train_pair_classifier.py` → `models/flu/{version}/runs/training_{bundle}_{ts}/` (per experiment)
- Shell wrappers: `scripts/stage2_esm2.sh`, `scripts/stage3_dataset.sh`, `scripts/stage4_train.sh`

- **DataSAIL C1 (cluster-based 1D) does not scale to our cluster granularity by design (2026-05-31)**: tried C1 as a head-to-head against our single-slot cluster_disjoint bicc on Flu A HA-NA. **The paper explicitly recommends K ≈ 50 clusters** — Joeres et al. 2025, "Effect of solvers and hyper-parameters, scalability" section around Fig. 6: "the quality of the splits does not improve for K > 150 and is already good for K ≈ 50, showing that a rather small number of clusters is sufficient". Fig. 6 only benchmarks up to K > 150 with a 2-hour per-solver timelimit. The "scalable heuristic" framing in the abstract refers to "cluster down to small K first, then ILP on K", NOT "ILP on N entities". Our use case requires K = our_native_cluster_count (e.g., 7,578 HA clusters at id095 on full corpus; ~750 even at 1000-isolate subsample) so the C1-vs-bicc comparison stays apples-to-apples — that's 50–1000× the recommended K. **Empirical scaling on our data**: 200 isolates / 98 clusters / id095 / ε=0.50 ran in 11s with 0 drops (supports the pre-registered "C1 1D doesn't have the bipartite mega-component drop problem" prediction). 1000 isolates / ~750 clusters / id100 / ε=0.50 hit >22 min in cvxpy `ConeMatrixStuffing` compile, never reached SCIP, 21 GB RAM. Projected at full corpus K=7,578: ~57M cvxpy.max nodes, terabytes of RAM. **Root cause**: C1's ILP objective in `datasail/solver/cluster_1d.py` is `Σ over cluster pairs (i,j) of weight × intra_sim × max(x[s,i] - x[s,j])` — O(K² × splits) cvxpy expressions; compile memory scales super-linearly. **Path B (precomputed distance matrix to bypass DataSAIL's internal clustering) verified working at small scale**, with three wrapper gotchas worth knowing for future C1 use: (a) the 1D technique string is `'C1e'` (constrain slot E) or `'C1f'` per `parsers.py:88` choices, NOT plain `'C1'` — plain `'C1'` is silently accepted but produces no assignments because per-mode preprocessing keys the result dicts by `'C1e'`/`'C1f'`, triggering `routine.py:164` short-circuit to `return None, None, None`; (b) `read_protein_data` calls `remove_duplicate_values` which collapses entities whose `e_data` values are identical — placeholder `e_data` MUST be unique per entity (using `seq_hash` as both key and value works since DataSAIL never runs mmseqs on the matrix path); (c) similarity-matrix input routes through `SpectralClustering` (which doesn't handle binary block-diagonal inputs cleanly — log shows all-1 statistics), distance-matrix input routes through `AgglomerativeClustering` (which DOES) — for recovering a precomputed binary block partition use the DISTANCE path with `linkage='complete'`. **Implication**: combined with the Phase 0 C2 finding (drops 51-71% of pairs structurally — separate root cause), no DataSAIL mode is operationally usable on full-corpus HA-NA Flu A. C2 fails by drop-strategy at any scale; C1 fails by ILP-blowup at our cluster granularity. A valid in-design DataSAIL run on our corpus would cluster down to K ≈ 150 internally and route 150 super-clusters — that IS scalable but compares different clusterings, not different routing algorithms on the same clustering. For alternative-to-LPT-greedy splitter design on our 7,578-cluster atom level, use `cluster_pair_weight_topk` rollup as input to a custom O(K × splits) algorithm (no off-the-shelf tool fits this regime). Branch: `feature/kfold-variance`. Script: `src/analysis/datasail_c1_haonly.py` (Path B, technique=C1e, distance-matrix path) — NOT committed; was a smoke test, not a full sweep.

## Considered but not yet pivoted: contig-level (nt_ctg) clustering
- Currently the pipeline clusters on two alphabets: aa (`prot_seq`) and nt = CDS-only (`cds_dna`, joined coding exons). A natural pivot would add nt_ctg = full contig DNA (includes UTRs) as a third clustering input.
- Source data is available: `genome_final.parquet` has `dna_seq` (full contig DNA per (assembly_id, replicon_type)), 868,240 rows = 8 segments × 108,530 isolates.
- Verified 2026-05-29: HA contig length stats (median 1715 nt, std 24.5, max 1817) vs HA CDS length stats (median 1701 nt, std 5.9, max 1713). UTR contribution at median is small (+14 nt, ~1%) but the LENGTH SPREAD is 4× wider (std 24.5 vs 5.9).
- **WRINKLE — assembly-artifact contamination of contig lengths**. The contig length variation is partly biological (UTR length variation) and partly artifactual (polyA tail inclusion in some assemblies but not others; primer trimming differences across submitters). Clustering on raw contigs as stored in `genome_final.parquet` is partly clustering on assembly-pipeline noise, not virus biology. For a "clean" contig clustering we'd want to trim to canonical Flu A segment boundaries (5' end of viral mRNA, 3' end before polyA) before clustering. This is a separate piece of work from the basic pivot. Mention this in `clusters.md` if/when we adopt nt_ctg as an operational alphabet.

### Phase 3 failure root cause + fixes (2026-04-02 → 2026-04-06)
**Problem**: Training ~15-34x slower per batch on Polaris vs Lambda (7.5 min/epoch vs 29s).
Three root causes identified:
1. **Memory explosion**: `KmerPairDataset` densified the ENTIRE 868K×4096 sparse matrix (14.2 GB) per fold. 4 concurrent folds = 56.9 GB → folds 1-11 OOM. Only fold 0 survived per pair.
2. **Cache thrashing**: 14.2 GB array >> L3 cache (~32-64 MB). Shuffled batch access = pure cache misses.
3. **Per-item overhead**: `self.pairs.iloc[idx]['label']` (pandas iloc) + `torch.tensor()` copy per item.

**Round 1 fixes** (in `src/models/train_pair_classifier.py`):
- Matrix subsetting: only densify rows used by this fold's pairs (~3.5 GB vs 14.2 GB). 4 folds fit: 14 GB vs 56.9 GB.
- Label pre-extraction: `self.labels = pairs['label'].values` — eliminates pandas iloc.
- Zero-copy tensors: `torch.from_numpy()` instead of `torch.tensor()` (safe with num_workers=0).
- tqdm `miniters=50` to reduce Lustre I/O from 1385 writes/epoch to ~28.
- `NUM_WORKERS` hard-coded to 0 (was configurable but num_workers>0 is 87% slower + incompatible with torch.from_numpy).
- Level 1 profiling added: per-epoch data_time/compute_time/eval_time in training_history.csv.

**Test run results (2026-04-06)**: Folds 0,1,3 ran 6 epochs before job died. Fold 2 OOM'd at model.to(device).
Profiling showed data loading is STILL 96-97% of epoch time (~460s data, ~4s compute, ~14s eval).
Matrix subsetting fixed memory but NOT speed. Two remaining issues:

**Round 2 diagnosis (2026-04-06)**:
1. **Fold 2 OOM**: TensorFlow installed in Polaris system conda env, loaded transitively by HuggingFace `transformers`
   (via esm2_utils.py). TF's default is to eagerly allocate all GPU memory. Added defensive env vars
   (`TF_CPP_MIN_LOG_LEVEL=3`, `TF_FORCE_GPU_ALLOW_GROWTH=true`) at top of script before any imports.
   Also added `torch.cuda.mem_get_info()` diagnostic before model.to(device) to capture GPU state.
2. **Data loading 460s/epoch**: Strongest hypothesis is `pin_memory=True` with 4 concurrent folds.
   pin_memory forces cudaHostAlloc (CUDA driver call that serializes across processes) for every batch.
   1385 batches/epoch × 4 folds = 5540 driver calls/epoch contending on same driver lock.
   The 18% speedup from pin_memory was benchmarked on Lambda (single fold) — likely counterproductive
   with 4 concurrent folds on Polaris. Added Level 2 diagnostic (micro-benchmark 10 batches with
